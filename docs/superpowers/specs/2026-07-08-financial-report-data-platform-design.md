# 财报事实库与证据库数据平台设计

日期：2026-07-08

状态：待用户审阅

## 1. 背景与目标

当前系统已经具备财报 PDF 上传、文本解析、预览、人工确认入库、基本面分析和个股档案展示能力。现有实现更接近“基本面仪表盘的数据补录工具”：可以解析部分三张表、业务分部、费用和少量年报扩展字段，但不足以稳定生成类似《绿的谐波》《金山办公》《石头科技》这类公司深度分析报告。

本设计的目标是将财报导入能力升级为“报告生成导向的数据平台”。系统不再只把 PDF 解析结果写入财务表，而是建立可追溯的事实库和证据库，支持财报数据、外部搜索资料、人工研究补录共同进入统一事实层，并最终支撑基本面深度分析、个股深度档案和公司深度报告草稿生成。

## 2. 设计原则

1. 财报导入只生产候选事实和证据，不直接覆盖正式结论。
2. 正式事实必须可追溯到财报页码、公告链接、网页来源或人工备注。
3. 财务数值以年报、公告、交易所和公司正式披露为强证据，普通网页搜索只作为候选线索。
4. 报告生成只消费已确认事实，并在数据不足时输出缺口清单，而不是生成无证据结论。
5. 数据模型同时支持强类型财务计算和行业差异较大的经营指标。
6. 保留当前接口兼容性，避免一次性重写基本面页面和个股档案。

## 3. 总体架构

系统采用“文档资产层、证据库、候选事实库、正式事实库、分析指标层、报告章节层”的分层架构。

```text
财报 PDF / 公告 / 网页搜索 / 公司官网 / 行业资料 / 人工补录
        ↓
文档资产层
        ↓
证据库
        ↓
候选事实库
        ↓
人工复核
        ↓
正式事实库
        ↓
分析指标层
        ↓
基本面深度分析 / 个股深度档案 / 深度报告章节
```

各层职责如下：

| 层级 | 职责 |
|---|---|
| 文档资产层 | 保存 PDF、公告、网页正文、搜索结果、解析作业、解析版本和抓取时间 |
| 证据库 | 保存原文片段、页码、URL、章节、来源等级、置信度和复核状态 |
| 候选事实库 | 保存解析或搜索得到的待复核指标值，支持冲突展示和人工修正 |
| 正式事实库 | 保存已确认、可用于计算和报告的结构化事实 |
| 分析指标层 | 计算增长、杜邦、利润影响、经营质量、估值分位和数据覆盖率 |
| 报告章节层 | 按目标报告结构组装章节、引用证据、输出缺口提示 |

## 4. 文档资产层

当前 `report_documents`、`report_parse_jobs`、`report_parse_results` 可以保留并扩展为文档资产层。新增能力应覆盖以下来源：

| 来源类型 | 示例 |
|---|---|
| annual_report | 年报、半年报、季报 PDF |
| announcement | 交易所公告、分红公告、回购公告、定增公告 |
| company_site | 公司官网、投资者关系页面 |
| industry_report | 行业机构、协会、统计机构、IDC、Gartner、艾瑞等 |
| broker_report | 券商研报 |
| news | 主流财经媒体 |
| search_result | 搜索结果摘要 |
| manual_note | 人工研究备注 |

文档资产层需要保存原始文件或正文、标题、URL、本地存储路径、发布时间、采集时间、来源类型、文件哈希、解析状态、解析版本和错误信息。

## 5. 证据库设计

新增 `evidence_items` 作为统一证据表。每条证据只表达“原文在哪里、说了什么、可信度如何”，不承担财务含义。

建议字段：

| 字段 | 含义 |
|---|---|
| id | 证据 ID |
| source_type | 来源类型，如 annual_report、company_site、industry_report、news |
| source_title | 年报标题、网页标题或公告标题 |
| source_url | 外部链接；本地 PDF 可保存本地路径 |
| source_date | 来源发布时间 |
| collected_at | 系统采集时间 |
| document_id | 关联文档资产 ID |
| code | 关联股票代码，可为空 |
| company_name | 公司名称，可为空 |
| topic | 业务、行业、专利、产能、用户、竞争、估值等 |
| snippet | 原文片段 |
| page_no | 财报页码，网页可为空 |
| section_name | 年报章节、表格名称或网页段落标题 |
| locator_json | 表格行列、段落索引、字符范围等定位信息 |
| confidence | 解析置信度 |
| trust_level | A、B、C、D 来源等级 |
| review_status | pending、confirmed、rejected |
| reviewer_note | 人工备注 |

来源等级规则：

| 等级 | 来源 | 使用规则 |
|---|---|---|
| A | 年报、公告、交易所、公司官网、投资者关系 | 可支撑正式事实和报告关键结论 |
| B | 权威行业机构、协会、统计机构、IDC、Gartner、艾瑞等 | 可支撑行业空间、市占率、排名 |
| C | 券商研报、主流财经媒体 | 可作为参考证据，需要标注来源 |
| D | 普通网页、自媒体、论坛 | 只作为线索，不直接支撑正式结论 |

## 6. 候选事实库

新增 `candidate_facts`。财报解析、外部搜索和人工补录都先写入候选事实，再由人工确认进入正式事实库。

建议字段：

| 字段 | 含义 |
|---|---|
| id | 候选事实 ID |
| code | 股票代码 |
| company_name | 公司名称 |
| period | 报告期或数据期 |
| fact_type | financial、segment、operation、industry、valuation 等 |
| metric_name | 指标名称 |
| metric_value | 指标值 |
| metric_unit | 元、万元、%、万人、台等 |
| dimension | 产品、地区、渠道、客户、项目等维度 |
| dimension_value | 维度值 |
| evidence_id | 主要证据 ID |
| evidence_ids_json | 多证据列表 |
| source_type | 来源类型 |
| trust_level | 来源等级 |
| confidence | 解析置信度 |
| parser_version | 解析器版本 |
| existing_fact_id | 可能覆盖的已有正式事实 |
| conflict_group | 同一指标多来源冲突分组 |
| review_status | pending、confirmed、modified、rejected |
| reviewer_note | 人工备注 |

候选事实工作台必须支持确认、修改、驳回、保留旧值和查看冲突来源。低置信度字段和空值不得静默覆盖正式事实。

## 7. 正式事实库

正式事实库采用“通用事实表 + 强类型明细表”的组合。

### 7.1 通用事实表

新增 `confirmed_facts`，承接所有已确认事实，尤其适合行业差异大的指标。

| 字段 | 含义 |
|---|---|
| id | 事实 ID |
| code | 股票代码 |
| company_name | 公司名称 |
| period | 报告期或数据期 |
| period_type | annual、semi_annual、quarter、date、range |
| fact_type | financial、segment、operation、industry、valuation 等 |
| metric_name | 指标名，如 WPS 月活用户、海外收入占比 |
| metric_value | 数值 |
| metric_unit | 单位 |
| dimension | 产品、地区、客户、渠道、项目等维度 |
| dimension_value | 具体维度值 |
| evidence_id | 主证据 ID |
| evidence_ids_json | 多证据列表 |
| source_type | 来源类型 |
| trust_level | 来源等级 |
| review_status | confirmed、superseded |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 7.2 强类型明细表

高频、要参与计算的内容沉淀为强类型表：

| 表 | 用途 |
|---|---|
| financial_statement_facts | 三张表核心财务数据，支撑增长、杜邦和现金流匹配 |
| profit_impact_facts | 收入、毛利、费用、减值、投资收益、税费等净利润影响因素 |
| business_segment_facts | 产品、行业、地区、销售模式分部，支撑业务分析和毛利贡献 |
| operation_quality_facts | 应收、存货、合同负债、收现率、周转率等经营质量指标 |
| business_metric_facts | 销量、产量、平均售价、单位成本、用户数、付费率、市占率等 |
| rd_project_facts | 研发投入、研发人员、研发项目、专利和技术储备 |
| capacity_project_facts | 募投、产能、在建工程、固定资产、资本开支 |
| capital_action_facts | 股本、资本公积、分红、回购、定增、股权激励、解禁 |
| valuation_market_facts | PE、PB、PS、PEG、市值、估值分位、同业对比 |

当前表的迁移定位：

| 当前表 | 新定位 |
|---|---|
| financial_reports | 迁移或映射到 financial_statement_facts |
| business_segments | 迁移或映射到 business_segment_facts |
| expense_items | 迁移或映射到 profit_impact_facts |
| annual_report_extractions | 拆分到多个事实域 |
| report_documents、report_parse_jobs、report_parse_results | 保留并扩展为文档资产层 |

## 8. 事实数据域清单

报告生成需要覆盖以下 10 个数据域。

1. 基础三张表数据：营业收入、营业成本、毛利、毛利率、归母净利润、扣非净利润、营业利润、利润总额、EPS、ROE、总资产、净资产、总负债、货币资金、经营现金流净额、销售收现、投资现金流、筹资现金流。
2. 利润影响因素：销售费用、管理费用、研发费用、财务费用、其他收益、投资收益、公允价值变动收益、信用减值损失、资产减值损失、资产处置收益、营业外收入、营业外支出、所得税费用、少数股东损益、政府补助、利息收入、汇兑损益。
3. 业务分部：按产品、行业、地区、销售模式入库收入、成本、毛利、毛利率、同比变化、收入占比、毛利占比。
4. 经营质量：应收账款、应收票据、合同资产、合同负债、预收款、存货总额、原材料、在产品、库存商品、发出商品、存货跌价准备、应收账款坏账准备、应收账款账龄、存货周转率、应收账款周转率、销售收现率。
5. 产品经营指标：销量、产量、库存量、平均售价、单位成本、单机毛利、渠道收入、线上收入、线下收入、海外收入、国内收入、海外毛利率、市场份额、行业排名。
6. 用户和商业模式指标：月活用户、付费用户、机构客户数、订阅收入、授权收入、广告收入、续费率、ARPU、云服务收入、AI 产品收入或商业化进展。
7. 研发、专利和技术储备：研发投入、研发投入占收入比例、研发人员数量、研发人员占比、研发项目名称、项目阶段、项目预算、当期投入、累计投入、专利总数、发明专利、实用新型、境内专利、境外专利、软件著作权。
8. 产能、募投和资本开支：在建工程、固定资产、资本开支、募投项目名称、计划投资、累计投入、项目进度、设计产能、新增产能、预计收益、产能利用率。
9. 资本结构和股本扩张：总股本、资本公积、未分配利润、每股资本公积、每股净资产、分红金额、分红率、回购金额、定增、可转债、股权激励、限售解禁、股本变动原因。
10. 估值和市场定位：PE、PB、PS、PEG、市值、股价区间、年度最高价、年度最低价、历史估值分位、同业公司估值、同业收入、同业利润、同业 ROE、行业空间、国产替代率、公司市场份额。

## 9. 外部搜索采集

外部搜索用于补充财报之外的信息，尤其是行业地位、市场空间、竞争格局、国产替代、用户指标、产品进展和同业资料。搜索结果不能直接进入正式事实库，只能生成外部证据和候选事实。

流程如下：

```text
搜索任务
  ↓
搜索结果列表
  ↓
网页、公告或资料正文抓取
  ↓
来源等级判断
  ↓
证据片段抽取
  ↓
候选事实生成
  ↓
人工复核
  ↓
正式事实库
```

优先场景：

| 场景 | 示例 |
|---|---|
| 行业地位 | 绿的谐波 谐波减速器 市占率 |
| 竞争格局 | 石头科技 扫地机器人 全球份额 IDC |
| 用户指标 | 金山办公 WPS 月活 付费用户 |
| 国产替代 | 谐波减速器 国产替代 进口替代 |
| 产品进展 | 金山办公 WPS AI 商业化 |
| 产能建设 | 绿的谐波 募投项目 产能 |
| 估值对比 | 可比公司 PE PB PS |

外部资料引用必须展示 URL、来源名称、发布时间和系统采集时间。过期资料不得悄悄升级为当前结论。

## 10. 数据覆盖率系统

新增数据覆盖率检查，用于判断每家公司、每个报告期、每个报告章节是否具备足够事实。

覆盖状态：

| 状态 | 含义 |
|---|---|
| confirmed | 已确认，可直接用于报告 |
| pending_review | 已解析或搜索到，需要人工复核 |
| missing | 系统没有数据 |
| external_required | 财报本身通常不提供，需要外部资料 |
| not_applicable | 该行业或公司不适用 |
| conflicted | 多来源冲突，需要人工选择 |

报告章节生成规则：

| 覆盖情况 | 行为 |
|---|---|
| 数据充分 | 生成完整章节、图表、结论和证据引用 |
| 数据部分缺失 | 生成章节并提示缺失项 |
| 只有弱证据 | 使用谨慎表述并标注来源等级 |
| 关键数据缺失 | 不生成强结论，只生成补录清单 |
| 数据冲突 | 阻塞该指标并要求人工选择来源 |

## 11. 工作流

### 11.1 财报导入流水线

1. 上传年报 PDF。
2. 识别公司、报告期、报告类型。
3. 提取页面文本、表格、章节标题和页码。
4. 抽取证据片段。
5. 生成候选事实。
6. 工作台按数据域展示候选事实和证据。
7. 用户确认、修改或驳回。
8. 已确认事实进入正式事实库。

### 11.2 外部搜索流水线

1. 用户或系统创建搜索任务。
2. 系统搜索并抓取网页、公告、行业资料。
3. 判断来源等级和发布时间。
4. 抽取证据片段和候选事实。
5. 展示多来源冲突。
6. 用户选择可信来源并确认。
7. 已确认事实进入正式事实库。

### 11.3 人工补录流水线

1. 用户录入指标、数据期、维度和值。
2. 绑定已有证据或填写人工备注。
3. 形成候选事实。
4. 用户确认后进入正式事实库。

### 11.4 报告生成流水线

1. 选择公司和报告期。
2. 运行数据覆盖率检查。
3. 计算分析指标。
4. 按章节模板组装内容。
5. 插入证据引用。
6. 输出报告草稿和缺口清单。

## 12. 报告章节模板

报告章节建议对应目标 Word 报告风格：

1. 行业逻辑与公司定位。
2. 公司业务分析。
3. 各财务环节对净利润影响。
4. 影响业绩的因素及排序。
5. 财务基础与股本扩张潜力。
6. 研发、专利和产能建设。
7. 国产替代、竞争格局或商业模式。
8. 市场定位与估值。

每个章节都必须声明依赖数据域、覆盖率状态和证据来源。章节生成优先输出 Markdown，后续再考虑 Word 或 PDF 导出。

## 13. 实施阶段

### 阶段 1：财报事实库 + 证据库底座

交付内容：

1. 新增 `evidence_items`、`candidate_facts`、`confirmed_facts`。
2. 财报 PDF 解析结果先进入候选事实，不直接覆盖正式表。
3. 每个候选事实绑定原文证据、页码、字段来源和置信度。
4. 当前 `financial_reports`、`business_segments`、`expense_items` 保持兼容。
5. 导入确认支持确认、修改、驳回、保留旧值。

验收标准：

1. 上传一份年报后，可以看到候选事实清单和证据来源。
2. 确认后的数据能追溯到 PDF 页码或原文片段。
3. 空值或低置信度字段不会静默覆盖已有正式数据。
4. 基本面页面和个股档案仍能正常读取旧接口。

### 阶段 2：补齐报告核心数据域

交付内容：

1. 强化业务分部、利润影响、经营质量、研发产能资本四类数据域。
2. 新增经营质量和研发产能相关强类型表或事实映射。
3. 在个股档案中展示核心证据和缺口。

验收标准：

1. 能生成公司业务分析的数据表和图表。
2. 能生成各财务环节对净利润影响的排序。
3. 能提示经营质量缺口，比如应收、存货、合同负债缺失。
4. 能在个股档案中展示研发、专利、产能、资本结构证据。

### 阶段 3：外部搜索采集

交付内容：

1. 新增搜索任务管理。
2. 保存搜索结果、网页正文、来源等级和发布时间。
3. 抽取候选事实和外部证据。
4. 同一指标多来源冲突时展示冲突。
5. 外部事实必须人工确认后才能进入正式事实库。

验收标准：

1. 可以针对某家公司创建搜索任务。
2. 搜索结果能沉淀为候选证据和候选事实。
3. 报告引用外部资料时显示 URL、来源、发布时间、采集时间。
4. D 级来源不能直接支撑正式结论，只能作为线索。

### 阶段 4：报告章节生成

交付内容：

1. 建立报告模板和章节依赖矩阵。
2. 每章先做数据覆盖率检查。
3. 数据充分则生成完整章节，数据不足则生成缺口清单。
4. 每个关键判断附带证据引用。
5. 支持导出 Markdown。

验收标准：

1. 对绿的谐波能生成业务分部、利润影响、研发专利、产能建设、估值定位草稿。
2. 对金山办公能生成 WPS 业务、用户指标、AI 进展、软件国产替代、估值定位草稿。
3. 对石头科技能生成产品销量、单价、海外业务、经营质量、市场份额草稿。
4. 任何缺数据的章节不强行编结论，而是明确列出缺口和建议搜索或补录项。

## 14. 当前系统落地判断

当前实现方向是正确的，但仍处于“导入 MVP + 基本面仪表盘”阶段。

已经具备：

1. PDF 上传和文档资产雏形。
2. 解析作业和解析结果保存。
3. 核心财务字段、业务分部、费用和部分年报扩展字段。
4. 基本面分析层的趋势、杜邦、估值分位、分部贡献和观察信号骨架。

主要缺口：

1. 没有独立证据库，正式表丢失字段级来源。
2. 没有候选事实库，解析结果容易过早进入正式表。
3. 没有覆盖率系统，报告生成无法系统性暴露缺口。
4. 经营质量、产品经营、用户指标、行业地位和外部估值对比数据不足。
5. 外部搜索尚未纳入可追溯研究资料层。
6. 人工复核页没有按报告数据域组织，也没有冲突处理和覆盖保护。

## 15. 风险与约束

1. 外部网页信息时效性强，必须保存发布时间和采集时间。
2. 普通搜索结果噪声高，不能绕过复核进入正式事实库。
3. PDF 表格解析存在误判，必须保留原文证据和人工确认。
4. 不同公司行业差异大，低频指标优先进入通用事实表，稳定后再沉淀强类型表。
5. 初期应优先保证事实链路可靠，而不是追求全自动生成完整报告。
6. 当前项目使用 SQLite，本地单用户场景可行；若后续多用户协作，需要补充权限、审计和迁移策略。

## 16. 非目标

1. 本阶段不承诺完全自动生成可直接发布的最终报告。
2. 本阶段不把券商研报观点当作公司事实。
3. 本阶段不使用普通网页数据覆盖正式财务数值。
4. 本阶段不一次性替换所有旧接口，而是通过兼容视图或映射逐步迁移。

## 17. 推荐优先级

优先级建议如下：

1. 先建设证据库、候选事实库和正式事实库。
2. 再改造财报导入确认流程，避免静默覆盖和来源丢失。
3. 再补齐业务分部、利润影响、经营质量、研发产能资本四类核心数据域。
4. 再加入外部搜索采集，补行业地位、竞争格局和用户指标。
5. 最后建设报告章节生成和 Markdown 导出。

这一路径可以让系统逐步从“财报解析工具”升级成“可追溯的公司深度研究平台”。
