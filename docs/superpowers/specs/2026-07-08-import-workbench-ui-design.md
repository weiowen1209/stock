# 财报导入与事实复核工作台 UI 设计

日期：2026-07-08

状态：已确认方向 A：研究流水线式

## 1. 设计目标

财报导入页不再只是“上传 PDF + 展示几个解析字段”，而是升级为财报事实生产线。页面需要让用户清楚看到：

1. 当前财报处于哪个处理阶段。
2. 系统解析出了哪些候选事实。
3. 每个候选事实来自哪段证据。
4. 候选值是否会覆盖已有正式值。
5. 哪些事实可以批量确认，哪些必须逐条复核。

本设计已确认采用“研究流水线式”布局，先服务阶段 1 的证据库、候选事实库、正式事实库闭环。报告章节式页面留到报告生成阶段再做。

## 2. 页面结构

页面分为三栏。

### 2.1 左侧：流程导航与文档资产库

左侧承担“我在哪里”的导航作用。

流程节点：

1. 上传或选择财报。
2. 解析候选事实。
3. 证据复核。
4. 确认入库。

文档资产库展示当前公司相关财报 PDF、报告期、解析状态和确认状态。阶段 1 只要求支持选择文档、查看预览和上传新财报，不要求做复杂文件管理。

### 2.2 中间：候选事实复核队列

中间是主工作区，展示候选事实列表和复核动作。

顶部展示当前公司、报告期和关键统计：

1. 已确认数量。
2. 待复核数量。
3. 冲突或风险数量。
4. 证据覆盖率。

候选事实按数据域筛选：

1. 全部。
2. 基础财务。
3. 利润影响。
4. 业务分部。
5. 经营质量。
6. 研发产能。

候选事实列表字段：

| 字段 | 含义 |
|---|---|
| 指标 | 指标名称、指标 key、所属数据域 |
| 候选值 | 解析值或人工修正值 |
| 来源等级 | A、B、C、D |
| 状态 | 待确认、需复核、冲突、已确认、已驳回 |
| 操作 | 确认、修改、驳回、选择来源 |

阶段 1 支持的动作：

1. 确认单条事实。
2. 修改候选值后确认。
3. 驳回候选事实。
4. 批量确认 A 级且高置信度事实。
5. 只看风险项。

### 2.3 右侧：证据与覆盖风险

右侧跟随当前选中的候选事实变化，用来回答“为什么相信这个值”和“确认后会发生什么”。

右侧展示：

1. 风险提示。
2. 原文证据片段。
3. 财报页码或来源 URL。
4. 命中标签和解析置信度。
5. 已有正式值。
6. 新候选值。
7. 覆盖动作建议。

阶段 1 的关键规则是：低置信度、冲突项、可能误识别项不得批量确认，必须逐条处理。

## 3. 视觉与交互原则

1. 页面优先表达“研究流水线”，不是传统后台表格。
2. 中间列表用于高效复核，右侧证据卡用于建立信任。
3. 风险项必须有明显视觉区分，例如浅橙或浅红背景。
4. 批量确认只对高可信事实开放，不能给用户造成“一键全信”的错觉。
5. 修改、驳回、确认都要保留操作状态，后续可扩展审计日志。
6. 空值不会覆盖旧值，这个规则需要在确认区明确展示。

## 4. 阶段 1 开发边界

阶段 1 要实现：

1. 左侧文档列表和当前解析状态。
2. 中间候选事实列表。
3. 候选事实按数据域筛选。
4. 右侧证据片段和覆盖风险展示。
5. 单条确认、修改、驳回。
6. 高可信事实批量确认。
7. 确认后写入正式事实库，并同步旧表保持现有页面兼容。

阶段 1 不实现：

1. 外部搜索采集。
2. 报告章节生成。
3. PDF 原文页内嵌高亮预览。
4. 多人协作审计。
5. 复杂版本回滚。

## 5. 后续演进

阶段 2 可以把经营质量、研发产能和资本结构的数据域补齐。

阶段 3 可以加入外部搜索采集，并在右侧证据卡展示 URL、发布时间和来源等级。

阶段 4 可以新增“报告章节式”入口，把已确认事实组装成深度报告草稿。

## 6. 已确认原型方向

已确认方向：A 研究流水线式。

原型结构：

```text
左侧：流程导航 + 文档资产库
中间：候选事实复核队列 + 风险筛选 + 批量确认
右侧：证据片段 + 覆盖旧值风险 + 修改/驳回
底部/确认区：只自动确认高可信事实，低置信度和冲突项逐条处理
```

后续进入开发前，任何涉及导入工作台布局或核心交互的变更，都需要先更新 UI 原型并让用户确认。
